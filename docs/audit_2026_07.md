# Front-end audit — July 2026

**What this is:** the tracked findings of a six-subsystem adversarial audit of
`pkgs/cli/`, run 2026-07-02 — after the Phase 1 resolution-correctness arc of
[lsp_v2.md](lsp_v2.md) and before Phase 2 (incremental engine). It catalogs
correctness issues, modelling gaps, and refactoring opportunities, and records
the sequence to address them in.

**Method.** Six parallel deep reviews (element/resolution, checker + inference,
codegen + encoder, loader/driver/CLI, LSP, lexer/parser/AST), each verified
against call sites; the checker, codegen, loader, and parser reviews
additionally **reproduced their top findings end-to-end** with `bin/hawk.sh
check`/`run`. Markers: **[confirmed]** = reproduced by execution; **[latent]** =
statically verified, needs an input (usually a name collision) the corpus lacks;
**[debt]** = design/modelling debt, no wrong answer today.

**What held up.** The Phase 1 machinery itself — per-file tables, `TypeId`,
`resolve_type_owner`, the T3/T4 owner-keying, `infer_callee_kind` as the single
callee authority — was traced at every site and found consistent. The encoder is
clean. The lexer is the healthiest component. The problems below are around
that core, not in it.

## Cross-cutting themes

1. **The checker has outright validation gaps** — whole categories of wrong
   programs check clean, then trap or silently miscompute (CH1–CH4, CH9).
2. **Values (consts, globals, natives) never got the owner-correct treatment
   functions and types got** — they are the surviving `global_functions`-class
   flat tables (CG3–CG5).
3. **Codegen lacks *lexical* scoping discipline**, not just file-level — block
   shadowing and const inlining miscompile with no collision needed (CG1, CG2).
4. **Builtin identity is a bare name string** (`Result`/`Option`/`List`/`Map`/
   `Iterator`/`Ordering`/`Error`) across checker, inference, and codegen — the
   T4 flip turned this from hygiene into a miscompile hazard (X1, CG6).
5. **Loader error paths and file identity** are the real Phase 2 blockers —
   silent failures, non-canonical identity, the import graph discarded
   (LD1–LD6, LD13).
6. **The LSP still runs on the flat resolution model the compiler retired**
   (LS1–LS4) — while the owner-correct inputs are computed and thrown away.

## Highlights

### Best bang for the buck

Small, mechanical, high payoff — most are one-function fixes over data that
already exists:

- **LS1** — thread `file_bare_surface`/`file_namespaces` into the LSP's
  `DocAnalysis` and rewrite `find_decl_site` over them. The owner-correct answer
  is already computed by the loader and discarded; fixes wrong-target
  hover/go-to-def and adds `ns.member` resolution with zero inference.
- **CH2** — type-check assignments: mirror the `Let` arm's `expect_type` call.
- **EL2** — the barrel-collision diagnostic: `LibraryNamespace.collisions` is
  already recorded, just never read.
- **CH-CN** — `collect_names`: add the missing `Let` arm (also unblocks CG4) and
  merge the `fn`/`const` buckets.
- **EL14** — `unify`: compare `TypeId` like `is_assignable` now does
  (one-liner, mirrors the T4 follow-up fix).
- **PA4** — brace-depth-aware `sync_to_decl`: kills phantom top-level decls and
  most recovery cascades.
- **LD2/LD3** — surface resolve/read failures in `link_imports` (the `None`
  arms currently drop silently).
- **X1 stopgap** — a reserved-builtin-name collision error (`Result`, `List`,
  …) buys time before the full TypeId-identity sweep — done (both, batch 4).
- **CH25** — diagnostics point at `expr_span(arg)` instead of the whole call:
  cheap, big LSP UX win.

### Basic correctness

The "checks clean, wrong at runtime" class — close these before investing
further in the analyzer's reputation:

- **CH1** match exhaustiveness (codegen *assumes* it; non-exhaustive matches
  silently take the last arm or trap).
- **CH2** assignment typing, **CH3** operator typing, **CH4** `if`/`else`
  branch unification.
- **CH9** struct-literal missing fields + unknown pattern variants: move from
  codegen asserts to checker diagnostics.
- **CG1/CG2** const-inlining scope capture and block-scope clobbering
  (miscompiles in ordinary single-library code) — fixed in batch 3.
- **PA1/PA2** interpolation: errors swallowed (the one place a syntax error
  ships to the runtime) and wrong spans inside `${...}`.
- **PA3** the `<` lookahead silently rewriting comparisons into generic calls.
- **LD6** check errors in imported files ship to bytecode.

### Fundamental modelling problems

Where the code doesn't represent a real source relationship — the audit's
"shortcuts that will bite again" list:

- **X2 (decided + enforced)** — the three-name-space model vs the kind-blind
  tables: resolved by spec change (one name space per scope); enforcement
  diagnostics landed with batch 2.
- **LD1/LD17** — no canonical file identity: raw path strings, three behaviors
  for three spellings of the same root. Nothing incremental can be built on
  today's identity story.
- **X1** — builtin semantics keyed by name string instead of `TypeId` against
  the core owner.
- **EL13 + PA11** — no symbol identity: elements carry no spans and no file;
  the AST is missing name-spans systematically. Blocks Phase 3 semantic
  rename/references outright.
- **LD13–LD15** — the import graph is built then flattened away; diagnostics
  exist in ~5 ad-hoc shapes with three attribution mechanisms (one keyed by the
  file's *source text*).
- **CH19** — uncontrolled re-inference: the checker re-infers each call's
  arguments up to five times via paths that can disagree (two confirmed
  false-positive bugs came from exactly this). The memoized per-node
  `infer_expr` the inference header already promises is also Phase 3's
  `type_at`.
- **EL11** — `''` as both "anonymous program" (a real table key) and "unknown
  owner" (a miss sentinel).

## Sequencing

Batches in order; the parser batch is independent and can land any time.

1. **Loader foundations** — **DONE 2026-07-02** (these *are* Phase 2
   prerequisites): canonical `FileId` minted at every entry point (LD1, LD17);
   loader surfaces resolve/read/check-of-import failures (LD2, LD3, LD6);
   `LoadedImports` keeps the import graph (LD13); one `Diagnostic` model keyed
   per file (LD4, LD7, LD15); overlay-aware file provider owned by the session
   (LD5).
2. **Checker-gaps batch** — **DONE 2026-07-02**: CH1–CH4, CH9, CH-CN, plus the
   one-name-space enforcement diagnostics from the X2 decision (see below) —
   each small, independent, with conformance tests.
3. **Value owner-keying + codegen scoping** — **DONE 2026-07-02**:
   consts/globals/natives get the T3/T4 playbook (CG3–CG5), block save/restore
   + const compiled in its own file/scope context via a scoped unit-context
   helper (CG1, CG2, CG-D1). Fixpoint-validated (byte-identical throughout —
   the corpus never exercised the broken paths).
4. **Builtin identity by TypeId** — **DONE 2026-07-03**: one sweep replacing
   `name == 'Result'`-class checks with an owner test across
   checker/inference/codegen (X1, CG6), plus the `Error`/`Args` floor entries
   (EL8). The reserved-name "stopgap" became a permanent spec rule (scoping.md
   "Reserved type names") alongside — not instead of — the identity sweep.
5. **The T4 tail in the checker** — **DONE 2026-07-03**: CH5 (batch 1), CH6,
   CH14, CH15, EL6, EL9 — "resolve via FileScope / pass the def you already
   hold" edits, validated by extending the `shared_type_name` conformance
   tests (collided generic `Pair<T>`). Rode along: EL14/EL16 (unify identity
   guard, contravariant fn params), CG7 (nested generic patterns), CH7/CH8
   (lambda expected-type false positives), and a batch-4 regression fix (the
   EL8 pin must allow same-owner re-registration — checking an SDK core file
   as primary registers it twice).
6. **Phase 3 enablers** (fold into Phase 3 as designed): memoized `infer_expr`
   (CH19), `SymbolId` + element/AST spans (EL13, PA11), LSP `find_decl_site`
   rewrite + overlay fix + UTF-16 boundary (LS1, LD5, LS6).
7. **Interface identity** — **DONE 2026-07-03**: the last "flat by design"
   area. Interface references (conformances, super-interfaces, type-parameter
   bounds) resolve in their declaring file's scope and are stored as `TypeId`s;
   `is_assignable`/`satisfies_bound`/conformance checking compare identities
   (closing CH24's display-string keying); codegen's interface tables
   (`interface_methods`, default-method units, `ImplInfo`) are owner-qualified.
   `impl ns.Iface for T` now resolves through the namespace (the parser had
   been silently *discarding* the qualifier — the SDK's `impl env.Env`/`impl
   time.Clock` only worked via the flat leak); `sdk/std/fs`+`process` migrated
   to `impl io.Reader …`. The hermetic floor gained real `Eq`/`Display`/`Debug` interface
   entries. Follow-up (same day): `Eq`/`Ord`/`Display`/`Debug` joined the
   reserved list outright — the pinned-but-declarable middle tier proved more
   confusing than useful (a user `Display` half-works with `${}` via
   selector dispatch). Conformance: `iface-identity` (collided `Shape`s:
   per-owner conformance, defaults, params, qualified impl),
   `iface-bound-identity` (a bound is satisfied by its own file's interface
   only), `type-reserved-names` extended.

- **Parser batch (any time)**: PA1–PA4; PA1/PA2 early — they're the only
  front-door soundness holes in the parser.

## Decision (taken 2026-07): one name space per scope

scoping.md previously promised three independent name spaces (a `type Set` and
a value `Set` coexist), but the implementation was kind-blind end to end:
`file_owned`, `file_bare_surface`, and the namespace surfaces pool
fn/type/const/let into one table (EL1), and `check_duplicates` split fn and
const into separate buckets so `fn max` + `const max` was accepted with
ask-order-dependent resolution (CH-CN).

**Decided: one name space per scope.** A scope may introduce a given name at
most once, across all declaration kinds — a same-file `fn max` + `const max`
(or `fn Config` + `struct Config`, or `import std.fs;` + `fn fs`) is a
duplicate-name check error, exactly like a same-kind duplicate.
[scoping.md](scoping.md) (Core model, File scope) is updated. Basis: the
corpus (stdlib, compiler, examples, tests) measured **zero** cross-kind name
reuse on 2026-07-02, so the rule was free; it matches the LLM-clarity goal
(one name, one meaning) and what the implementation already does. The
kind-blind surface/ownership tables are therefore the *model*, not a bug.

Note it does **not** replace owner-correctness: same-name-across-libraries
stays legal and still needs the per-owner machinery; this rule only removes
the within-scope ambiguity class.

**Remaining work (enforcement — batch 2):**
- [x] same-file cross-kind duplicates are errors: merge `collect_names`'s
  `fn`/`const` buckets into one value-space set, add the missing `Let` arm,
  and include type/interface names and import-namespace bindings in the one
  set (CH-CN; closes EL1's same-file arm).
- [x] cross-kind (and same-kind) collisions introduced by the bare surface —
  two `as _` imports, or `as _` vs prelude — get the collision diagnostic
  (EL4); barrels already have recorded collisions awaiting a reader (EL2).
- [x] conformance tests: `fn`+`const`, `fn`+`struct`, decl+import-namespace,
  duplicate top-level `let` (`mod-one-name-space`, `mod-bare-collision`).

# Findings

## Checker + inference (`checker/checker.hawk`, `element/inference.hawk`)

- [x] **CH1 [confirmed]** No match exhaustiveness check — `checker.hawk:850`
  checks arm-type agreement only; codegen (`codegen.hawk:942,995`) assumes
  exhaustiveness and treats the last arm as always-matching. Non-exhaustive
  match silently returns the wrong arm; with payloads, extracts fields from the
  wrong variant and traps. docs/language.md speaks as if enforced.
- [x] **CH2 [confirmed]** Assignments never type-checked — `checker.hawk:524`
  passes the target type only as an inference hint; `x = 'a'` on `mut x = 1`,
  `xs[0] = 'a'` on `List<Int>`, and mut-field corruption all pass. Fix mirrors
  the `Let` arm's `expect_type`.
- [x] **CH3 [confirmed]** Binary/unary operators never type-checked —
  `checker.hawk:695` recurses only; `inference.hawk:163` returns the left
  operand's type. `1 + 'a'` checks clean, traps at runtime.
- [x] **CH4 [confirmed]** `if`/`else` branch types never unified —
  `checker.hawk:781`, `inference.hawk:324`. `let x = if c { 1 } else { 'a' }`
  checks clean. Match arms *are* unified; `if` is the inconsistent sibling.
- [x] **CH5 [confirmed, collision]** `impl_self_type` stamps `self` from the
  flat table — `checker.hawk:1387` (flat `named_type`); `check_conformance`
  fetches interface + target flat and resolves `interface_args` via the flat
  `resolve_type_ref` (`checker.hawk:344,352,367`). Under a collided type name:
  false `no such field` on real fields, wrong element as the conformance
  method source. Fix: resolve via `file_scope(...).resolve_type_owner` like
  `infer_struct`. _Fixed (pulled forward from batch 5 — LD6's closure checking
  made it live: checking `libpoint_a`/`libpoint_b` as closure members hit the
  false positive exactly as predicted): `impl_self_type` mirrors codegen's
  owner-correct `named_self_type`; `check_conformance` resolves the impl
  target + `interface_args` scope-aware. Interface *names* stay flat (the
  documented deferral)._
- [x] **CH6 [confirmed, collision]** `receiver_bindings` owner-blind —
  `inference.hawk:1410` refetches the receiver's type params via flat
  `resolve_type_def_by_name` though every caller already holds the
  owner-correct def; under collision, member checks silently vanish
  (unbound `T` → TypeParameter leniency). _Fixed: takes the receiver's
  `TypeId`, resolves via `resolve_type_def`. Conformance: `shared_type_name`
  extended with a collided generic `Pair<T>`._
- [x] **CH7 [confirmed]** False positive: lambdas in `if`-expression tails —
  `check_block` (`checker.hawk:500`) checks tails with `expected = None` and
  the `If` arm drops `expected`; inference threads it. "cannot infer lambda
  parameter" on code inference types fine. _Fixed: `check_block` gains
  `tail_expected`, threaded from the Block/If expression arms; codegen's
  `emit_block_value`/`if_expr` mirror it (its lambda units stashed Unknown
  params otherwise). Conformance: `lambda_infer_context`._
- [x] **CH8 [confirmed]** False positive: lambda args to static methods —
  `expected_arg_types` (`inference.hawk:787`) has no static-receiver path
  (`Type.method`, `ns.Type.method`) though `resolve_static_receiver` exists.
  _Fixed: it now consults `resolve_static_receiver` (subsuming the old
  `ns.fn` half). Conformance: `lambda_infer_context`._
- [x] **CH9 [confirmed]** Checker/codegen diagnostic split — missing
  struct-literal fields (`codegen.hawk:1405`) and unknown pattern variants
  (`codegen.hawk:1202`) are validated only in codegen: `check` says clean, the
  LSP shows nothing, `emit`/`run` fails. Move to checker; codegen keeps asserts.
- [x] **CH-CN [confirmed]** `collect_names` misses top-level `let` entirely
  (`checker.hawk:118`) — same-file duplicate `let x` passes (enables CG4) —
  and splits `fn`/`const` buckets so `fn max` + `const max` is accepted. The
  resolver's `top_decl_name` *does* include `Let`; the lists have diverged.

- [ ] **CH10 [latent]** Generic instantiation consistency unchecked —

- [ ] **CH21 [debt]** `check_lambda` re-implements `infer_lambda`'s cascade

- [ ] **CH22 [debt]** Double resolution: `method_resolves` then

- [ ] **CH23 [debt]** Parallel `scope`/`muts` maps + `(errors, scope, ctx,

## Codegen (`codegen/codegen.hawk`, `codegen/module_scope.hawk`, encoder)

- [x] **CG1 [confirmed]** Const inlining compiles the initializer in the
  *consumer's* scope — `emit_const` (`codegen.hawk:452`) runs `expr(init)`
  with the consumer's `slots`/`type_scope`/`self.file`. A local shadowing a
  name the initializer references is captured (`const a = b + 1` + local

  `emit_const` compiles inside `enter_file_context`/`restore_file_context`

- [x] **CG2 [confirmed]** No block scoping in codegen — `declare_local`

  the entire statement (`:1191,1391`) — phantom nodes for offset queries.
- [ ] **PA7 [latent]** `Stmt.Expr`/`Stmt.Assign` spans cover only the first
  token (`parser.hawk:973,992`) — inconsistent with Return/Throw/Let; wrong
  for statement-level LSP features.
- [ ] **PA8 [debt]** `else if` synthesizes unmarked wrapper Blocks
  (`parser.hawk:1280,1317`) — indistinguishable from source `else { if }`
  except by span-text sniffing (the heuristic class MatchOrigin retired).
  Wants a synthetic marker.
- [ ] **PA9 [hygiene]** `import '${x}'` treated as literal text
  (`parser.hawk:374`) — no error, confusing "module not found" later.
- [ ] **PA10 [latent]** `parse_impl_generics` silently mis-lowers a generic
  argument as a type param (`parser.hawk:719`) — `impl Box<Indexed<T>>` drops
  the args, fabricates `TypeParam{Indexed}`, no diagnostic anywhere.
- [ ] **PA11 [debt]** Systematic missing name-spans — `FieldDef` (no span at
  all), `FieldInit`, `Expr.Field`'s member name, `CallArg.label` (computed
  then discarded, `parser.hawk:1685`), `ImplDecl.interface_name`,
  `InterfaceDecl.super_interfaces`, `TypeParam.bounds`, `ImportDecl`
  path/alias, `NamedType.namespace` (token discarded, `:622`). The Phase 3
  rename blocker. Recommend an `Identifier {name, span}` struct adopted
  incrementally. Pair with EL13.
- [ ] **PA12 [debt]** Stringly/overloaded AST shapes — Binary/Unary `op` is
  span text (synthesized for `<<`, `:1548`); `Decl.Type` serves struct +
  native type via `is_native`; `Pattern.Literal` wraps arbitrary exprs
  (interpolated-string "literal patterns" parse, `:1459`).
- [ ] **PA13 [hygiene]** Grammar/doc drift — `\u{…}` listed as "not yet in the
  grammar" but implemented; empty `struct E {}` accepted vs grammar's ≥1
  field; interface methods accept bodies vs `methodSig`; `{-1: x}` map-literal
  corner errors confusingly. (Precedence table, trailing commas, optional-`;`
  all verified matching.)
- [ ] **PA14 [hygiene]** Lexer nits — CRLF: comment spans keep the trailing
  `\r` (`lexer.hawk:206`, matters for fmt re-emission); a `//` inside `${…}`
  silently eats the interpolation tail (disappears with PA2's fix);
  describe.hawk renders string parts without re-escaping (breaks the one-line
  diff-key property).
